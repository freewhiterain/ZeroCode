# ZeroCode 学习导读

> 这份文档是为「第一次读这个项目」的你准备的地图。配合代码里 `# 【讲解】`
> 前缀的注释一起看（那些是专门写给你的学习注释，和项目原有注释有区分）。

---

## 一、这个项目是什么

ZeroCode 是一个**终端 AI 编程助手**——你在终端里和它聊天，它能读写文件、
执行命令、帮你写代码。本质上是一个自制版的 Claude Code。

一句话概括它的工作原理：

> **把「大模型 API」和「一堆本地工具」用一个循环粘起来。**
> 模型说"我要读某个文件" → 程序执行并把内容发回去 → 模型接着思考 →
> 再要下一个工具……直到模型不再要工具、给出最终回答。

这个循环就叫 **Agent 循环**，代码在 `zerocode/agent.py` 的 `Agent.run()`，
是全项目的心脏。

## 二、一条消息的旅程（核心数据流）

假设你在界面里输入"帮我看看 config.py 写了什么"，会发生：

```
你敲回车
   │
   ▼
app.py（Textual 界面）把文字交给 Agent
   │
   ▼
conversation.py  把它记入对话历史（list[Message]）
   │
   ▼
agent.py  Agent.run() 开始循环 ──────────────────────┐
   │  ① 组装 system prompt（prompts.py）              │
   │  ② 把对话历史序列化成 API 格式（serialization.py）│
   │  ③ 调用 LLM（client.py → 网络请求）              │
   ▼                                                 │
模型流式回复："我来读一下" + 工具调用 ReadFile(config.py)
   │                                                 │
   ▼                                                 │
agent.py  收到工具调用：                              │
   │  ④ 权限检查（permissions/）——读文件默认放行      │
   │  ⑤ 参数校验（pydantic）→ 执行 tools/read_file.py │
   │  ⑥ 把文件内容作为 tool_result 写回对话历史        │
   └──────── 回到 ① 再次调用 LLM ────────────────────┘
   │
   ▼（模型这次没有再要工具）
模型给出最终回答 → LoopComplete 事件 → 循环结束，界面显示完整回复
```

整个过程中，Agent 不直接碰界面——它把发生的每件事包装成 **AgentEvent**
（"模型输出了一段字"、"工具执行完了"、"需要用户授权"……）yield 出来，
由 app.py 决定怎么画到屏幕上。**事件流是理解这个项目的第一把钥匙。**

## 三、推荐阅读顺序

**全项目 90+ 个源文件现在都加上了 `# 【讲解】` 学习注释**（按 11 批陆续完成）。
下面这个顺序是"从心脏到四肢"的建议路径，不代表文件优先级——你完全可以
跳着看感兴趣的子系统。

### 第 1 批：核心主线 ★ 从这里开始

| 顺序 | 文件 | 看什么 |
|---|---|---|
| 1 | `zerocode/__main__.py` | 程序怎么启动、各组件怎么组装（依赖注入的雏形） |
| 2 | `zerocode/tools/base.py` | Tool 基类 + 流式事件定义，很短，先建立"工具"的概念 |
| 3 | `zerocode/conversation.py` | 对话历史长什么样、token 怎么估算 |
| 4 | `zerocode/agent.py` | **精读 `Agent.run()`**，对照注释里的步骤①~⑤走一遍 |
| 5 | `zerocode/client.py` | 精读 `AnthropicClient.stream()`，看 API 流事件怎么翻译 |

读完这 5 个文件，你就理解了项目 80% 的运行机制。

### 第 2 批：让主线丰满起来

| 文件 | 内容 |
|---|---|
| `zerocode/tools/` | 全部工具实现（read/write/edit/bash/glob/grep/ask_user/agent_tool/team_*/task_* 等）+ 注册表 `__init__.py` + 延迟工具 `impl/tool_search.py` |
| `zerocode/serialization.py` | 内部 Message → 各家 API 格式的翻译 |
| `zerocode/prompts.py` | system prompt 是怎么一段段拼出来的 |
| `zerocode/config.py` / `validator.py` | 配置加载与校验 |
| `zerocode/permissions/` | 权限系统：modes.py（模式矩阵）→ rules.py（规则）→ dangerous.py（危险命令）→ sandbox.py（路径沙箱）→ checker.py（责任链合并判定） |

### 第 3 批：进阶子系统

| 子系统 | 说明 |
|---|---|
| `context/manager.py` | 上下文压缩，两层防御（大输出落盘替换 + LLM 摘要压缩），全项目算法最密集的文件 |
| `agents/` | 子 agent：定义解析（parser/loader）、fork 机制、工具过滤、后台任务管理、调用链追踪 |
| `memory/` + `hooks/` | 自动记忆提取/检索、会话持久化；生命周期钩子引擎 |
| `teams/` | 多 agent 团队协作：文件信箱通信、共享任务看板、进程内/tmux/iTerm2 三种队友后端 |
| `skills/` + `mcp_client/` + `filehistory/` + `worktree/` | 技能包（含动态工具加载）、MCP 协议客户端、/rewind 版本控制、git worktree 隔离 |
| `commands/` | 斜杠命令系统：注册表、解析器、UI 补全、14 个内置命令处理器 |
| `app.py` | Textual 界面层（77KB，最后看）。重点读 `_send_message` 里的
  `async for event in self.agent.run(...)`——这是事件流从 Agent 到 UI
  的唯一接口 |

## 四、关键概念小词典

| 概念 | 解释 |
|---|---|
| **Agent 循环** | "模型要工具 → 执行 → 回报结果 → 再问模型" 的 while 循环 |
| **事件流 / yield** | Agent 用异步生成器边运行边吐事件，UI 层 `async for` 消费；逻辑与界面解耦 |
| **tool_use / tool_result** | LLM API 的硬性配对规则：模型的每个工具调用请求，下一条消息里必须有对应结果 |
| **context window** | 模型一次能"看到"的最大 token 数。对话超长就要压缩（compact） |
| **prompt cache** | 请求前缀逐字节相同时 API 只收 10% 费用；所以稳定内容（system、工具表）要放前面且保持不变 |
| **token 锚点** | 用 API 返回的真实用量做基准，只对之后新增的消息做字符估算——比全量估算准 |
| **pydantic** | 数据校验库。工具参数模型既生成给模型看的 JSON Schema，又校验模型传回的参数 |
| **asyncio.Future** | "待填写的答案栏"。权限弹窗的实现核心：Agent await 它睡觉，用户选择后 set_result 唤醒 |
| **system-reminder** | 程序想对模型说的话，借 user 角色 + 特殊标签传入 |

## 五、常用 Python 语法速查（本项目高频出现）

- `@dataclass` — 自动生成 `__init__` 等方法的类，用来定义纯数据结构
- `async def` / `await` — 异步函数：等网络/磁盘时把 CPU 让给别人
- `async def` + `yield` — **异步生成器**：边执行边产出值，本项目的骨架模式
- `A | B`（类型注解中） — 联合类型，"A 或 B"
- `ABC` / `@abstractmethod` — 抽象基类：定义接口，强制子类实现
- `field(default_factory=list)` — dataclass 里可变默认值的正确写法
- `asyncio.gather(*tasks)` — 并发执行多个异步任务并收集全部结果
- 函数内 `import` — 延迟导入，加快启动或避免循环依赖

## 六、动手建议

1. **跑起来**：在 `ZeroCode/` 下执行 `uv run zerocode`（需要先在
   `.zerocode/config.local.yaml` 配好 API key）。用的时候开着
   `.zerocode/debug.log` 对照看日志。
2. **加打印**：在 `Agent.run()` 里加几行 `log.info(...)`，观察每轮循环
   的事件顺序，比光读代码印象深十倍。
3. **写个工具**：模仿 `tools/glob.py` 写一个自己的工具（比如"统计目录下
   代码行数"），在 `tools/__init__.py` 注册，然后让模型调用它——完成这个
   练习就算真正入门了。
4. **跑测试**：`uv run python -m pytest tests/test_agent.py -q`。
   （注：`test_message_splicing` 和 `test_concurrent_batch_execution`
   两个用例在当前仓库本来就失败——它们假设项目根目录存在 README.md。）

## 七、已知小问题（读代码时不要被迷惑）

- `ZEROCODE.md` 写着"技术栈: Go"，实际是纯 Python 项目，那个文件像是占位内容。
- 代码里 `.ZeroCode` 和 `.zerocode` 两种目录写法混用。Windows 文件系统不区分
  大小写所以能跑，但这是个隐患（Linux 上会变成两个不同目录）。
- **测试与代码有多处漂移**：`tests/` 目录里不少用例引用了当前 `zerocode/`
  代码里不存在的函数/参数/常量（比如 `zerocode.memory.session` 缺
  `build_time_gap_message`、`WorktreeManager.__init__` 缺 `file_cache`
  参数、`zerocode.prompts` 缺 `BASE_PERSONA`、`commands/handlers/` 缺
  `do.py`）。这些失败在加注释之前就已存在（每批改动后都用
  `git stash` 前后对比验证过，均与本次注释无关），说明测试代码和实现
  代码不是同步维护的——读测试文件时留意，不要以为看到的是"当前真实行为"。

## 八、CodeExplore：从符号直接读取源码和调用关系

CodeExplore 是内置的 Python 代码图工具。它借鉴了 CodeGraph、GitNexus 和
Graphify 的两个核心思路：用语法树提取真实代码符号，以及把符号之间的调用
关系保存成可查询图。第一版刻意保持本地、轻量和保守，不启动后台服务，也不
把无法唯一确定的同名调用硬连成边。

### 使用方法

首次在项目中使用时，在 ZeroCode 输入：

```text
/graph init
```

索引保存在项目的 `.zerocode/codegraph/graph.json`，它是生成物，已被
`.gitignore` 忽略。之后模型可以调用：

```json
{
  "symbols": ["create_default_registry", "Agent.run"],
  "max_depth": 1
}
```

其中 `symbols` 接受 1～3 个精确函数、方法或类名；`max_depth` 为 0～2，
0 只返回当前源码，1 或 2 会继续返回有界的 caller/callee 关系。常用管理命令：

| 命令 | 作用 |
|---|---|
| `/graph status` | 查看文件数、符号数、边数和解析问题数 |
| `/graph refresh` | 主动执行一次增量刷新 |
| `/graph rebuild` | 原子化重建索引；失败时保留旧索引 |

CodeExplore 每次查询前也会做一次懒刷新：先用文件大小和修改时间快速判断，
再对疑似变化的文件计算 SHA-256；内容没变就不重新解析。修改文件后无需手动
重建整个项目。

### 建议阅读顺序

| 顺序 | 文件 | 看点 |
|---|---|---|
| 1 | `zerocode/codegraph/models.py` | 符号、调用事实、边和文件状态的数据契约 |
| 2 | `zerocode/codegraph/parser.py` | Tree-sitter Query 如何提取类、函数、调用和导入 |
| 3 | `zerocode/codegraph/resolver.py` | 类作用域、同文件、显式导入、全局唯一四级解析 |
| 4 | `zerocode/codegraph/store.py` | NetworkX 封装、单文件 JSON 和原子替换 |
| 5 | `zerocode/codegraph/indexer.py` | Git 文件扫描、全量构建和哈希增量刷新 |
| 6 | `zerocode/codegraph/formatter.py` | 当前源码、关系遍历、节点/输出上限 |
| 7 | `zerocode/tools/code_explore.py` | 工具参数校验、异步线程卸载和错误降级 |
| 8 | `zerocode/commands/handlers/graph.py` | `/graph` 管理命令如何接入 UI |

解析器先产出 `CallFact`，解析器不会立刻猜目标。resolver 只有在证据能确定唯一
目标时才生成 `calls` 边；歧义事实仍随索引保存，便于将来增强解析算法。这是
本实现最重要的准确性边界。
